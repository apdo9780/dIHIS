## Implementation Phases

### Phase 1: Database Models ✅

**File**: `app/models/icu.py`

Two new SQLAlchemy ORM models support clinical telemetry and alerts:

#### VitalSign Model
```python
class VitalSign(db.Model):
    """Stores patient vital signs with continuous telemetry."""
    - heart_rate (int): BPM
    - blood_pressure_systolic (int): mmHg
    - blood_pressure_diastolic (int): mmHg
    - oxygen_saturation (float): SpO2 %
    - temperature (float): Celsius
    - respiratory_rate (int): breaths/min
    - recorded_at (datetime): Measurement timestamp
    - Foreign Key to Patient
```

**Features**:
- Indexed on `patient_id` for fast lookups
- Indexed on `recorded_at` for time-series queries
- Automatic timestamps
- Cascade relationships

#### ClinicalAlert Model
```python
class ClinicalAlert(db.Model):
    """Tracks alerts generated by ICU AI."""
    - alert_level (str): 'Warning' or 'Critical'
    - message (str): Alert description
    - recommended_intervention (str): Clinical action
    - vital_sign_id (int): Reference to triggering vital
    - is_acknowledged (bool): Clinician acknowledgment
    - acknowledged_at (datetime): When acknowledged
    - acknowledged_by (str): Clinician identifier
```

**Features**:
- Indexed on multiple fields for efficient querying
- Acknowledgment tracking for workflow
- Bidirectional relationships
- Audit trail capabilities

**Database Registration**: Both models automatically imported and registered in `app/models/__init__.py`

### Phase 2: AI Agent Logic ✅

**File**: `app/ai_agents/icu_specialist.py`

The `ICUSpecialistAI` class provides pure business logic, intentionally separated from database operations for testability.

#### Core Medical Thresholds

```python
NORMAL_RANGES = {
    'heart_rate': {'min': 60, 'max': 100},
    'systolic_bp': {'min': 90, 'max': 140},
    'diastolic_bp': {'min': 60, 'max': 90},
    'oxygen_saturation': {'min': 95, 'max': 100},
    'temperature': {'min': 36.5, 'max': 37.5},
    'respiratory_rate': {'min': 12, 'max': 20}
}

CRITICAL_THRESHOLDS = {
    'heart_rate_critical_low': 40,
    'heart_rate_critical_high': 140,
    'systolic_bp_critical_low': 80,
    'oxygen_saturation_critical': 85,
    'temperature_critical_high': 40,
    'temperature_critical_low': 35,
    'respiratory_rate_critical_low': 8,
    'respiratory_rate_critical_high': 30
}
```

#### Key Methods

1. **`evaluate_vitals(vital_data: Dict) → Dict`**
   - Analyzes 6 vital parameters
   - Evaluates against medical thresholds
   - Identifies abnormalities with severity levels
   - Returns structured evaluation with findings
   - Individual evaluation methods for each vital parameter

2. **`generate_alerts(evaluation_results: Dict) → List[Dict]`**
   - Converts evaluation findings into clinical alerts
   - Maps abnormalities to evidence-based interventions
   - Handles complex multi-parameter conditions
   - Returns actionable alert messages

3. **`recommend_intervention(condition: str) → str`**
   - Maps clinical conditions to specific interventions
   - Provides step-by-step clinical protocols
   - Covers: Hypoxia, Hypertension, Hypotension, Tachycardia, Bradycardia, Fever, Hypothermia

#### Design Principles

- **Pure Logic**: No database coupling for easy testing
- **Evidence-Based**: Medical thresholds from clinical guidelines
- **Modular**: Each vital sign evaluated independently
- **Extensible**: Ready for LLM integration in future phases

#### API Endpoints

##### 1. Record Vital Signs
```
POST /icu/api/vitals
Content-Type: application/json

{
    "patient_id": 1,
    "heart_rate": 92,
    "blood_pressure_systolic": 128,
    "blood_pressure_diastolic": 82,
    "oxygen_saturation": 96.5,
    "temperature": 37.2,
    "respiratory_rate": 16,
    "recorded_at": "2024-01-15T14:30:00"  # Optional
}

Response:
{
    "success": true,
    "vital_sign_id": 42,
    "vital_sign": { ... },
    "evaluation": {
        "is_normal": true,
        "overall_severity": "Normal",
        "abnormalities": []
    },
    "alerts_generated": [],
    "alert_count": 0,
    "message": "Vital signs recorded. All parameters normal."
}
```

**Features**:
- Automatic vital sign ingestion
- Real-time AI evaluation
- Automatic alert generation
- Comprehensive response with evaluation results

##### 2. Get Alerts
```
GET /icu/api/alerts?acknowledged=false&alert_level=Critical&limit=50

Response:
{
    "success": true,
    "count": 2,
    "alerts": [
        {
            "id": 1,
            "patient_id": 5,
            "alert_level": "Critical",
            "message": "Severe Hypoxemia Detected",
            "recommended_intervention": "...",
            "is_acknowledged": false,
            "created_at": "2024-01-15T14:35:00"
        }

}
```

**Query Parameters**:
- `acknowledged` (bool): Filter by acknowledgment status
- `patient_id` (int): Filter by specific patient
- `alert_level` (str): 'Warning' or 'Critical'
- `limit` (int): Max records (default 50)

## Database Schema

### vital_signs Table
```sql
CREATE TABLE vital_signs (
    id INTEGER PRIMARY KEY,
    patient_id INTEGER NOT NULL FOREIGN KEY,
    heart_rate INTEGER NOT NULL,
    blood_pressure_systolic INTEGER NOT NULL,
    blood_pressure_diastolic INTEGER NOT NULL,
    oxygen_saturation FLOAT NOT NULL,
    temperature FLOAT NOT NULL,
    respiratory_rate INTEGER NOT NULL,
    recorded_at DATETIME NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_patient_id,
    INDEX idx_created_at
)
```

### clinical_alerts Table
```sql
CREATE TABLE clinical_alerts (
    id INTEGER PRIMARY KEY,
    patient_id INTEGER NOT NULL FOREIGN KEY,
    alert_level VARCHAR(20) NOT NULL,
    message TEXT NOT NULL,
    recommended_intervention TEXT NOT NULL,
    vital_sign_id INTEGER FOREIGN KEY,
    is_acknowledged BOOLEAN DEFAULT FALSE,
    acknowledged_at DATETIME,
    acknowledged_by VARCHAR(100),
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_patient_id,
    INDEX idx_alert_level,
    INDEX idx_is_acknowledged,
    INDEX idx_created_at
)
```

## Medical Evidence Basis

All vital sign thresholds and interventions are based on:

- **ACLS Guidelines** (American Heart Association)
- **Surviving Sepsis Campaign** bundles
- **ARDS Protocols** (Acute Respiratory Distress Syndrome)
- **Cardiogenic Shock** management
- **ICU Care** standards

### Critical Thresholds Rationale

| Parameter | Critical Low | Critical High | Clinical Significance |
|-----------|--------------|---------------|----------------------|
| Heart Rate | <40 BPM | >140 BPM | Life-threatening arrhythmias |
| Systolic BP | <80 mmHg | N/A | Shock risk |
| SpO2 | <85% | N/A | Severe hypoxemia |
| Temperature | <35°C | >40°C | Organ dysfunction risk |
| Respiratory Rate | <8 | >30 | Respiratory failure |

## Troubleshooting

### Dashboard Not Updating
- Check browser console (F12) for JavaScript errors
- Verify `/icu/api/dashboard-data` endpoint is accessible
- Clear browser cache and reload

### Alerts Not Generating
- Verify vital signs are crossing thresholds
- Check `app.db` exists and has data
- Ensure patient_id exists in patients table
- Review ICUSpecialistAI threshold values

### Performance Issues
- Check database indexes
- Monitor query times in Flask debug mode
- Consider reducing polling interval
- Check server resources

### API Errors
- Verify JSON payload format
- Ensure patient exists before recording vitals
- Check Content-Type header is `application/json`
- Review Flask debug logs
